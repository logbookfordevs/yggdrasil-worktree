# Product

## Register

brand

## Users

Yggtree serves developers, AI-assisted coding users, and tool builders who need to work across multiple Git branches without disturbing an active checkout. They are often context switching between feature work, bug fixes, reviews, experiments, and agent-driven changes.

The public site and docs should help a reader understand the workflow quickly, trust the safety model, and get to the right command without reading the entire CLI reference first. When a surface is tool-like, such as an interactive UI, settings flow, or dense command reference, product-style clarity should override brand expression.

## Product Purpose

Yggdrasil Worktree turns Git worktrees into a first-class workflow. It creates, opens, bootstraps, lists, and removes isolated worktrees so each task can live in its own clean environment.

Success means a developer can keep current work intact, start or resume another branch confidently, run risky experiments in a sandbox, and bring only the useful result back.

## Brand Personality

Practical, mythic, careful.

The brand can borrow from the Yggdrasil metaphor, but it should stay anchored in concrete developer workflows. The visual tone should feel distinctive enough for a public CLI site while keeping command examples, safety notes, and install paths easy to scan.

## Anti-references

- Generic SaaS docs pages with interchangeable cards, soft gradients, and vague productivity promises.
- Terminal-only hacker aesthetics that make the product feel narrower than the workflow it supports.
- Decorative mythology that obscures the actual Git and worktree behavior.
- Dense command references that present completeness before the reader has a working mental model.

## Design Principles

- Teach the workflow before the flag list.
- Make safety visible whenever a command can move, copy, apply, or delete work.
- Let the Yggdrasil metaphor add identity, not ambiguity.
- Prefer concrete examples over abstract claims.
- Keep public pages expressive, but make tool-like areas behave like tools.

## Accessibility & Inclusion

Use WCAG AA as the baseline for public site and docs work. Maintain readable contrast on dark surfaces, preserve keyboard and screen-reader navigation for docs, and respect reduced-motion preferences. Do not rely on color alone to communicate safety, state, or destructive behavior.
